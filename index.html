1.1.1 Definition

Cloud computing is the on-demand availability of computing resources (like servers, storage, databases, networking, software) over the internet. Crucially, this happens without direct active management by the user, and it's almost always based on a **pay-as-you-go** pricing model. It allows users and companies to access data and applications from virtually anywhere with an internet connection. These services are offered and managed by cloud service providers like Amazon Web Services (AWS), Microsoft Azure, and Google Cloud Platform (GCP).

The Official NIST Definition Deconstructed:

  • "Ubiquitous, convenient, on-demand network access...": It's everywhere, it's easy, you get it when you want it, and you access it over the internet.
  • "Shared pool of configurable computing resources...": A massive collection of servers, storage, etc., that can be programmatically assigned and shared among many users (multi-tenancy).
  • "Rapidly provisioned and released with minimal management...": You can scale up for a traffic spike and then scale back down just as quickly, with minimal effort.

1.1.2 Essential Characteristics

On-Demand Self-Service

Users can provision resources like server time or storage automatically, without needing to email an IT administrator or wait for manual fulfillment.
Analogy: Using a self-checkout at a grocery store instead of waiting for a cashier.

Broad Network Access

Services are accessible via standard network mechanisms (e.g., browsers, APIs) and from a wide range of devices (laptops, phones).
Analogy: Any standard web browser can access Google Docs, you don't need a special "Google Docs browser."

Resource Pooling

The provider's resources are pooled to serve multiple customers in a multi-tenant model. Resources are dynamically assigned based on demand. You don't know the exact physical server your code is on, and you don't need to.
Analogy: Residents in an apartment building share the building's main water supply, but each has their own secure apartment and tap.

Rapid Elasticity

Resources can be scaled up or down rapidly, often automatically. To the user, the supply appears unlimited.
Analogy: Your electricity supply. You can turn on one light or all the appliances in your house; the grid handles the load without you pre-booking the power.

Measured Service

Usage is metered, controlled, and reported, providing transparency for both the provider and consumer. You pay for what you use.
Analogy: A taxi meter that only charges for the distance you travel.

1.2.1 Relevant Technologies

Virtualization

This is the core enabling technology. It abstracts physical hardware, allowing a single server to run multiple independent operating systems or applications. This is the foundation of resource pooling and IaaS.

Service-Oriented Architecture (SOA)

An architectural approach where applications are built from distinct, independent services. This allows for scalability and integration, which is how massive cloud platforms are built.

Grid and Utility Computing

Grid computing involves connecting multiple computers to solve a common problem. Utility computing is the concept of providing computing as a metered service, like electricity. Cloud combines both ideas.

Distributed Computing

Spreading a single application or task across multiple computer systems. This is essential for the redundancy, fault tolerance, and scalability of the cloud.

API-based Access

APIs (Application Programming Interfaces) provide a flexible, programmable way to access and manage cloud services, enabling automation and the "Infrastructure as Code" (IaC) movement.

1.2.2 Cloud Service Delivery Models

IaaS (Infrastructure as a Service)

Provides virtualized hardware resources like VMs, storage, and networks. It offers the most control and flexibility. Examples: Amazon EC2, Google Compute Engine, Azure VMs.

  • Pros: Customizable, complete control, pay-per-use.
  • Security Spotlight: You are responsible for almost everything: securing the VM operating system, patching, network access rules, and monitoring.

PaaS (Platform as a Service)

Provides a platform and environment to develop, test, and deploy applications, so developers don't have to worry about the underlying infrastructure. Examples: Google App Engine, Heroku, Azure App Services.

  • Pros: Speeds up development, scalable, cost-efficient.
  • Security Spotlight: A shared responsibility. The provider secures the platform, but you must secure your application code, dependencies, and configurations.

SaaS (Software as a Service)

Provides ready-to-use software applications over the internet, with no need to install or manage anything. Examples: Gmail, Microsoft 365, Zoom.

  • Pros: Easy to use, reduces management overhead, subscription-based.
  • Security Spotlight: You have the least control. Your primary concerns are data confidentiality, user access control (who can use it), and ensuring the provider meets your compliance needs.

1.2.3 Cloud Deployment Models

Public Cloud

Services offered by third-party providers (AWS, Azure) over the public internet to multiple organizations. Highly cost-effective due to shared infrastructure. Use Case: Websites, email services, online storage for non-sensitive data.

Private Cloud

Infrastructure used exclusively by a single organization. Can be hosted on-premises or externally. Offers the greatest control, customization, and security. Use Case: Financial institutions, government agencies with strict data sovereignty rules.

Hybrid Cloud

A combination of public and private clouds, allowing data and applications to move between them. Offers flexibility, scalability, and security. Use Case: Storing sensitive customer data on a private cloud while using a public cloud for a scalable e-commerce website.

Community Cloud

Infrastructure is shared by several organizations with common concerns (e.g., compliance, mission). Use Case: A group of hospitals sharing a cloud environment that is HIPAA compliant for patient records.

1.3 Introduction to Cloud Security

Cloud security is a collection of policies, technologies, applications, and controls that protect cloud-based systems, data, and infrastructure. It's a sub-domain of cybersecurity, but with unique challenges due to the dynamic and on-demand nature of cloud environments. The most fundamental concept to understand is the **Shared Responsibility Model**.

The Shared Responsibility Model

This model defines the division of security obligations between the cloud service provider (CSP) and the customer. The CSP is always responsible for the security **OF** the cloud, while the customer is always responsible for security **IN** the cloud.

You (Customer): Security IN the Cloud
  • Data
  • Applications
  • Identity & Access
  • Operating System
  • Network Controls
Provider (CSP): Security OF the Cloud
  • Compute & Storage Hardware
  • Databases (Physical)
  • Networking (Physical)
  • Physical Datacenters

The line of responsibility moves up or down depending on the service model (IaaS, PaaS, SaaS).

2.1 Data Identification & Classification

This foundational process involves two key steps: discovering where your data is and then labeling it based on its sensitivity and importance.

Data Identification

The process of discovering and recognizing sensitive or critical data. It answers: "Where is my data?" This includes locating data stored on-premises, in various cloud services (S3, RDS, etc.), and on user endpoints.

Data Classification

The process of organizing data into categories for easier protection. It answers: "How important is this data?" Classification is usually based on sensitivity, compliance needs (GDPR, HIPAA), and business impact.

Purpose & Benefits

The primary goal is to enable better risk management. By knowing what data you have and how sensitive it is, you can prioritize security measures, apply appropriate access controls, and ensure you meet regulatory compliance.

2.2 Example Data Classification Levels

Public

No damage if disclosed. E.g., Marketing material, public web content.

Internal Use Only

Moderate risk if exposed. E.g., internal policies, operational data.

Confidential

Could cause harm. E.g., employee records, project plans.

Restricted

Severe impact if leaked. E.g., financial records, trade secrets, encryption keys.

2.3 Data Asset Management in Cloud – Tagging

Tagging is the practice of attaching metadata (in key-value pairs) to your cloud resources. It acts like a label maker for your entire cloud environment, linking your classification efforts to actual resources.

# Example Tags for an S3 Bucket
{
  "Environment": "Production",
  "Owner": "HR-Department",
  "Data-Classification": "Confidential",
  "Cost-Center": "HR-123"
}

Tags help with:

  • Organizing Resources: Grouping assets by project or application.
  • Identifying Ownership: Quickly see which team is responsible for a resource.
  • Automating Policies: For example, "Deny public access to any resource tagged 'Data-Classification: Confidential'."
  • Tracking Cost: Filter billing reports by tags to understand costs per department.

2.4.1 Tokenization

Tokenization replaces sensitive data with a non-sensitive equivalent, a "token," that has no exploitable value outside its system. The real data is stored securely in a separate, highly protected database called a token vault.

  • Benefits: Reduces exposure of sensitive data, helps achieve compliance (like PCI-DSS), simplifies access controls.
  • Limitation: The security of the entire system depends on the security of the token vault.

2.4.2 Encryption: The Three States of Data

Data In Motion

Data traveling across networks (e.g., sending an email). Protection Methods: SSL/TLS (HTTPS), VPN tunnels, Secure APIs.

Data At Rest

Data stored on disks or in databases. Protection Methods: AES-256 encryption, disk encryption (BitLocker), cloud-native services (AWS KMS, Azure Key Vault).

Data In Use

Data being processed in memory (RAM). This is the most challenging state to protect. Protection Methods: Confidential computing, secure enclaves (Intel SGX), homomorphic encryption.

Security Spotlight: Key Management & DLP
  • Key Management: Encryption is only as strong as the security of its keys. Use a dedicated Key Management Service (KMS) like AWS KMS or Azure Key Vault to create, store, and rotate encryption keys securely.
  • Data Loss Prevention (DLP): These tools scan cloud environments to discover sensitive data and can automatically block unauthorized attempts to exfiltrate (move) it outside your secure perimeter.
  • Data Sovereignty: This is the concept that data is subject to the laws of the country in which it is located. When using the cloud, you must ensure your data is stored in regions that comply with your legal and regulatory requirements.

3.1 Types of Cloud Assets

An "asset" is any resource or component used in your cloud environment. Proper management involves inventorying and securing these core building blocks.

3.1.1 Compute Assets

Assets responsible for running applications and processing data. Examples: VMs (EC2), containers (Kubernetes), serverless functions (Lambda). Security: VM image hardening, regular patching, using security groups (firewalls).

3.1.2 Storage Assets

Assets that provide persistent data storage. Examples: Object storage (S3), block storage (EBS), file systems (EFS). Security: Encryption (at-rest/in-transit), access permissions, versioning, backups.

3.1.3 Network Assets

Assets that connect your cloud systems securely. Examples: VPCs, subnets, firewalls, load balancers, gateways. Security: Network segmentation, restrictive firewall rules, IP whitelisting.

3.2 Asset Management Pipeline & Leaks

This pipeline represents the lifecycle of a cloud asset. "Leaks" are vulnerabilities or misconfigurations that can occur at any stage.

Procurement
Processing
Tooling
Finding
3.2.1 Procurement Leaks

Mistakes made during asset acquisition/setup. Examples: Provisioning a VM with an outdated, unpatched image; creating an S3 bucket that is public by default. Prevention: Use approved, secured templates (Infrastructure as Code).

3.2.2 Processing Leaks

Vulnerabilities during the usage phase. Examples: An application bug that leads to data exposure; an admin granting overly permissive access. Prevention: Regular audits, strong IAM policies, runtime protection tools.

3.2.3 Tooling Leaks

Flaws in the tools used for management or monitoring. Examples: A CI/CD pipeline with exposed secrets; an insecure third-party monitoring tool. Prevention: Secure your toolchains, use secrets management.

3.2.4 Finding Leaks

Challenges in discovering or inventorying all assets. Example: "Shadow IT," where a developer spins up a server for a test project without telling anyone, and it gets forgotten but remains active. Prevention: Use cloud asset discovery tools and a CMDB.

3.3 Tagging Cloud Assets

Just as with data, tagging assets is crucial. It helps organize, track, secure, and optimize resources. A consistent tagging strategy is a hallmark of a mature cloud environment.

Best Practices for Tagging
  • Define a Global Tagging Policy: Create a standard set of required tags (e.g., `owner`, `environment`, `cost-center`) and make it company policy.
  • Automate Tagging: Use Infrastructure as Code (IaC) tools like Terraform or CloudFormation to enforce tagging on resource creation.
  • Enforce Through Policies: Use services like AWS Config Rules or Azure Policy to identify or even terminate non-compliant (untagged) resources.

3.4 Cloud Security Posture Management (CSPM)

CSPM is an emerging category of security tools that continuously monitor cloud environments for misconfigurations and compliance risks. In a large, dynamic cloud environment, it's impossible to check everything manually. CSPM automates this process.

What CSPM Tools Do:
  • Discover Assets: Continuously scan your cloud accounts to find all assets.
  • Detect Misconfigurations: Compare your asset configurations against a baseline of security best practices (e.g., CIS Benchmarks, NIST).
  • Alert on Risks: Automatically generate alerts for issues like public S3 buckets, overly permissive IAM roles, or unencrypted databases.
  • Enable Remediation: Some tools offer automated remediation to fix common issues.

4.1 Life Cycle for Identity and Access

The IAM lifecycle is the end-to-end, automated process of managing a user's identity and access from their first day to their last, ensuring security and least privilege.

Identity Creation & Provisioning

When a user joins, a unique identity is created and initial access is provisioned based on their role.

Maintenance

As a user changes roles or projects, their access is updated accordingly (granting new permissions, revoking old ones).

De-provisioning

When a user leaves, their access is immediately and completely removed to prevent unauthorized access.

4.2 Request, Approve

This is the formal process for granting access. It creates an audit trail and ensures accountability.

1

User: Requests access to a specific system or data.

2

Manager/Owner: Reviews the request based on role and need, then approves or denies it.

3

System: Automatically provisions the approved access.

The approval workflow can be automated and depend on the user's role, the sensitivity of the resource, and whether access is temporary or permanent.

4.3 Create, Delete, Grant, Revoke

These are the four fundamental actions in IAM systems that are performed by administrators (or automated systems).

Create/Delete

Refers to the identity itself. Creating a new user account or deleting it when they leave.

Grant/Revoke

Refers to permissions. Granting a user access to a specific database or revoking it when no longer needed.

4.4 Authentication

Authentication is the process of verifying a user is who they claim to be. It answers the question: "Who are you?"

Security Spotlight: MFA is Non-Negotiable

In modern security, MFA is considered a baseline requirement, not an optional extra. Compromised credentials are the #1 cause of data breaches, and MFA is the single most effective control to prevent this.

Common Methods
  • Password-based: The most common, but weakest form.
  • Multi-Factor Authentication (MFA): Something you know (password) + something you have (phone app) or something you are (fingerprint).
  • Biometric: Fingerprint, facial recognition.
  • OAuth/OpenID: Using a third-party identity (e.g., "Login with Google").

4.5 Authorization

Once authenticated, authorization determines what actions the user is allowed to perform. It answers the question: "What can you do?" It is the key to enforcing the Principle of Least Privilege.

Popular Models
  • Role-Based Access Control (RBAC): Access is granted based on a user's role (e.g., "Developer," "Auditor"). This is the most common model in cloud environments.
  • Attribute-Based Access Control (ABAC): Access is granted based on attributes of the user, resource, and environment (e.g., "Allow access if user is a manager AND it's business hours AND data is not PII").
The Evolution: Zero Trust

Zero Trust is a modern security model built on the philosophy of "never trust, always verify." It assumes that threats exist both outside and inside the network. Access is granted on a per-session basis, is strictly enforced, and requires re-authentication frequently, regardless of where the user is located.

4.6 Revalidate

Periodic revalidation (or attestation) of access rights ensures users only have the access they still need. This process is crucial for security and compliance, as it prevents "privilege creep"—where users accumulate unnecessary permissions over time.

The Revalidation Process
  1. A report is generated showing who has access to what.
  2. A system owner or manager reviews the list.
  3. They validate that each user's access is still required for their job.
  4. Any outdated or risky permissions are revoked.
